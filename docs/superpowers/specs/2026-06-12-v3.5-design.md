# OpenLeads v3.5 — "make the leads actually work" design

**Date:** 2026-06-12
**Branch:** `feat/v3.5`
**Status:** approved-by-default (autonomous build per user's standing preference)

## Why

v3.1 shipped an honest engine, but in practice the leads are a mess:

- **YC (the default source) returns all-`risky`, score-40 pattern guesses** and, in
  `verified_only` mode, took **26 s to produce 1 lead** on a normal network. Root
  cause: ground-truth site harvesting is gated behind `--deep` (off by default), so
  the single biggest accuracy lever never fires; and every founder triggers a live
  SMTP probe that hangs on a blocked port 25 (2 hosts × connect-timeout per lead).
- **HN returns good ground-truth leads but with junk**: sentence-fragments as company
  names, personal Gmail addresses and `jobs@`/`careers@` role mailboxes all labelled
  `safe`.
- Scoring is so pessimistic when port 25 is blocked (the common case) that legitimate,
  corroborated guesses never reach `safe` — so the tool looks empty.

The user's #1 requirement: **leads that are real, high-quality, deliverable, and the
best results possible** — Apollo/Hunter-grade, but free, keyless, local-first. Plus:
on-device send-time automation, IMAP "Sent" visibility, a free OpenRouter chatbot for
natural-language campaign setup, and a Claude-Code-inspired CLI. Ship as v3.5 to
GitHub + PyPI + npm.

## Design constraints (unchanged invariants)

- **Core stays stdlib-only**, zero runtime deps. `[chat]`/`[campaign]` extras optional.
- **Free + keyless by default.** Every feature must work at $0 with nothing configured.
- **Honesty.** We never relabel a blind guess as verified. But we *do* surface a
  calibrated confidence and stop being needlessly empty.
- Python 3.8+ compatible.

## 1. Lead-generation overhaul (the heart)

### 1.1 Port-25 reachability is probed once, not per-lead
A process-level, cached probe (`emails/netcheck.py`) checks outbound port 25 once. If
blocked, the SMTP verifier is skipped entirely instead of timing out per candidate.
This alone removes the multi-minute stalls. `doctor` already detects port 25; we reuse
the same check and memoize it.

### 1.2 Ground-truth site harvesting is **default-on**, deduped per domain
The biggest quality lever. For any corporate-domain lead we scrape the company's own
public pages (`/`, `/contact`, `/about`, `/team`, `security.txt`) for `@domain`
addresses — cheap, cached, and run **once per domain per run** (not per person). This
mirrors Hunter's domain-search:

- A harvested real address that matches our candidate → **exact ground truth → safe**.
- *Any* harvested address **teaches the domain's local-part pattern**, so every other
  person at that domain is built from an *observed* pattern, not a blind guess.

`--deep` still adds the heavier GitHub commit-email harvest and multi-candidate Gravatar.

### 1.3 Domain-grouped pipeline so the flywheel compounds within one run
`engine.build_leads` is restructured to resolve **per domain**: harvest the domain's
ground truth + learn its pattern first, then resolve every person at that domain
against the learned pattern. A pattern observed for one founder immediately upgrades
their co-founders. (Today learning happens but cross-pollination is racy across
parallel windows.)

### 1.4 Calibrated, honest scoring (Apollo/Hunter-style confidence)
`emails/score.py` gains a `confidence_pct` (0–100) and recalibrated tiers:

- **Observed pattern** (a sibling address at the domain was actually seen) + healthy
  infra (MX agreement, SPF/DMARC) → **safe** even with port 25 blocked. This is
  legitimate: the pattern is evidence, not a guess.
- **Common pattern** (`first` / `first.last`) on a professionally-hosted domain
  (Google/Microsoft) with SPF+DMARC → **risky but high** (a reachable, probable hit),
  surfaced with its percentage rather than buried.
- Free/personal mailbox guesses and role mailboxes stay gated exactly as before.

The `safe`/`risky`/`bad` contract is preserved; CSV/JSON gain a `Confidence %` column.

### 1.5 Quality cleanup
- HN company-name parser strips trailing role/marketing sentences and rejects names
  that are obviously a sentence (too many words / verbs).
- Personal-provider addresses are only `safe` when they're exact ground truth (a real
  published apply address), never a guess; they're tagged `personal`.
- Role/apply mailboxes (`jobs@`, `careers@`) are kept (they're real, intended-for-
  contact) but clearly tagged `role` so the user can filter.

### 1.6 New source: `domains` (Hunter-style company search)
A keyless source that takes a domain (or comma-separated domains, or a company name it
resolves to a homepage) and returns the harvested + pattern-derived contacts. Makes
"find emails at acme.com" a first-class path. 100% deliverability-focused.

**Acceptance:** YC `verified_only` returns multiple `safe` leads in < ~15 s on a normal
network; default `find` returns a mix clearly labelled by confidence; no sentence-as-
company-name rows; existing tests stay green.

## 2. Emails land in the provider's Sent folder (IMAP APPEND)

After a successful SMTP send, optionally `APPEND` the raw RFC822 message to the
mailbox's **Sent** folder over IMAP so it shows in the user's mail client.

- `outreach/providers.py`: `append_to_sent(raw_bytes, when, cfg)` that discovers the
  Sent mailbox via IMAP `LIST` special-use (`\Sent`), falling back to provider-known
  names (`[Gmail]/Sent Mail`, `Sent`, `Sent Items`, `Sent Messages`).
- New setting `save_to_sent` = `auto` | `always` | `never` (default `auto`). `auto`
  appends for every provider **except Gmail/Workspace**, which already journal SMTP
  sends to Sent (so we avoid duplicates).
- `sender.send_drafts` appends on a `sent` result; failures are logged, never fatal.

**Acceptance:** with IMAP creds, a live send appears in Sent; Gmail isn't duplicated;
no-IMAP users are unaffected.

## 3. On-device send-time automation

### 3.1 Send-time planner (`automate/sendtime.py`, pure)
Given drafts + policy + `now`, assign each a send time inside the next allowed window:
- Business-hours windows (default 08:00–11:00 & 13:00–16:00), weekdays only.
- Per-recipient timezone inference from country/city (offline lookup table) so mail
  arrives in *their* morning; unknown → sender's local tz.
- Spacing + jitter between sends; daily cap = warmup allowance.

### 3.2 Real OS scheduling (`automate/scheduler.py` upgraded)
- macOS: generate **and install** a launchd plist (`~/Library/LaunchAgents/…`) that
  runs the daily drip; `load`/`unload` helpers.
- Linux: install a crontab line.
- New CLI `openleads schedule [--at HH:MM] [--off]` to set up / tear down on-device
  automation, and `openleads schedule status`.

**Acceptance:** `openleads schedule --at 9:00` installs a working agent; `--off`
removes it; planner unit tests cover windows, tz, caps, jitter determinism (seeded).

## 4. Local chatbot (free, OpenRouter) for natural-language campaigns

`openleads/assistant.py`: interpret a free-form command like *"send 50 emails for my
SaaS to fintech founders at 9am"* into a structured `Action` (find + write + schedule),
then configure it. Uses a free OpenRouter model when a key is set; falls back to a
rule-based interpreter (extends the existing `intent` parser with count/time/goal
extraction) so it still works at $0.

- A small, explicit JSON action schema (`{intent, query, count, send_time, campaign,
  context}`) — the model fills it; we validate + execute.
- Wired into the chat REPL: action-intents ("send …", "schedule …", "set up …") route
  to the assistant; search-intents keep today's behavior. Always shows what it
  configured and asks to confirm before any live send.

**Acceptance:** "send 50 emails for X at 9am" configures a campaign + schedule
(dry-run preview first); works rule-based with no key; LLM path validated/guarded.

## 5. CLI overhaul (Claude-Code-inspired, brand colors)

Keep the monochrome + red-accent palette. Add:
- A polished boxed welcome/intro, a status line (source · count · deliverable · mailbox
  · port-25 · LLM), and a command palette.
- Streaming find with a live-updating spinner + per-lead lines and a final summary bar.
- Slash-command help styled like Claude Code; consistent section rules and color keys.
- `openleads` (no args) → the upgraded REPL; new `schedule` and `assistant` verbs.
- Graceful stdlib fallback preserved (no hard rich/ptk dependency in core).

**Acceptance:** the CLI looks cohesive and branded; non-TTY/`NO_COLOR` stays clean;
no new hard dependencies in the core.

## 6. Ship v3.5

- Version → `3.5.0` (pyproject, npm/package.json, `__init__`). CHANGELOG + README +

- PR to `main`, merge, tag `v3.5.0` to trigger the existing release pipeline
  (GitHub Release always; PyPI/npm publish gated on the user's tokens/OIDC).

## Testing

Every new pure module gets unit tests (planner windows/tz/caps, assistant parsing,
score calibration, Sent-folder discovery, HN name cleanup, port-25 memoization).
Network is never required in tests. Full suite + `ruff` must stay green; `/code-review`
before merge.

## Risk / mitigations

- **Over-claiming deliverability:** mitigated by keeping the honest tiers and only
  promoting to `safe` on *observed* evidence (ground truth or observed sibling pattern).
- **Site scraping latency:** cached + deduped per domain + short timeouts + parallel.
- **IMAP duplication on Gmail:** `auto` mode skips Gmail/Workspace.
- **Scope:** core invariants (stdlib core, keyless, $0) are non-negotiable and gate
  every feature.
